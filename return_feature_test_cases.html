Critical
Scenario
Two staff open the same invoice return modal at the same time (two tabs, two devices). Both read returnable = 10. Both submit return qty = 10. Both pass validation. Both restock 10 meters.
Risk
20 meters restocked from a 10-meter sale. 2× ledger credit given. Inventory inflated. Customer gets double refund. This is a real money loss.
// Fix — use DB-level locking in transaction: BEGIN TRANSACTION; -- Lock the invoice row for this transaction only: SELECT * FROM invoices WHERE id = ? FOR UPDATE; -- Now re-check returnable qty INSIDE the transaction: SELECT COALESCE(SUM(qty), 0) FROM invoice_returns WHERE invoice_id = ? AND item_id = ?; -- Only proceed if still returnable -- Other concurrent request will wait until this commits COMMIT;
EXPECT First request succeeds. Second request gets 422 — "Already returned" because qty was rechecked inside locked transaction.

Most important single fix: Change your Promise.all (one API call per item) to a single POST /api/invoices/{id}/return with an array of all items in one request body. One DB transaction, all-or-nothing, no partial state possible. This fixes the race condition, the partial failure, and the double-click problem all at once. Everything else is secondary to this architectural change.
